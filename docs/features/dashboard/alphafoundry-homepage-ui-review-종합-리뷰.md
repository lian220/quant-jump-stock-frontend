# alphafoundry.app 메인 화면 종합 리뷰

> 작성일: 2026-03-10
> 대상: `https://alphafoundry.app`
> 검토 범위: 관련 문서 8건 + 실제 서비스 화면(비로그인/로그인, PC/모바일 관점)
> 테스트 계정: `lian_test`

## 결론 요약

현재 메인 화면의 가장 큰 문제는 `메인 홈의 역할 정의 충돌`보다도 먼저, `뷰포트가 달라져도 모바일형 IA가 거의 그대로 유지되는 점`이다. 실제 점검에서 브라우저 폭을 넓혀도 햄버거 메뉴와 하단 탭바가 유지됐고, 로그인/비로그인 모두 첫 화면이 모바일 셸처럼 보였다.

문서 기준으로는 로그인 홈을 개인화 대시보드로 강화해야 한다는 합의가 이미 존재하지만, 실제 화면은 아직 `모바일 셸 + 제한된 요약 카드 + 기존 섹션 나열`에 가까웠다. 즉, 문서가 지적한 문제의식은 타당하지만, 현재 프로덕션에서는 `반응형 구조`, `정보 위계`, `로그인 보상 경험`이 모두 충분히 반영되지 않은 상태다.

## 주요 발견 사항

### P0. 데스크톱에서도 모바일형 정보구조가 유지됨

- 브라우저 폭을 넓힌 상태에서도 상단은 `햄버거 메뉴`, 하단은 `탭바` 중심 구조로 보였다.
- 로그인/비로그인 모두 `데스크톱용 전개형 네비게이션`이나 `멀티 컬럼 구조`가 첫 화면에서 드러나지 않았다.
- 결과적으로 사용자는 큰 화면에서도 서비스가 `모바일 앱 셸을 억지로 늘려 놓은 화면`처럼 느낄 가능성이 높다.

### P0. 첫 화면의 레이아웃 균형이 불안정함

- 로그인 홈 첫 화면에서 제목, CTA, 요약 카드가 좁은 세로 레이아웃 안에 갇혀 보였다.
- 넓은 화면에서도 좌우 공간 활용이 거의 되지 않아, `데이터 대시보드`라기보다 `모바일 카드 스택` 인상이 강했다.
- 문서에서 제안한 `PC multi-column` 원칙과 실제 경험이 가장 크게 어긋나는 지점이다.

### P1. 로그인 전후의 구조 차이는 있으나 로그인 보상이 충분히 크지 않음

- 비로그인 홈은 히어로 + CTA 중심, 로그인 홈은 `오늘의 AI 분석` 요약 카드와 CTA로 바뀐다.
- 하지만 로그인 후에도 핵심적으로 눈에 띄는 변화는 `상단 한 블록` 수준이며, 이후 섹션은 AI 종목, 뉴스, 전략 순으로 유사하게 이어진다.
- 문서가 강조한 `개인화 대시보드` 체감은 아직 약하다.

### P1. Dashboard API 기반 개인화가 첫 화면의 주인공이 아님

- 문서상 핵심 과제는 `subscriptions`, `signals`, `market`, `aiUsage` 노출이지만, 실제 홈에서는 개인 요약의 깊이가 제한적이다.
- 로그인 가치가 `로그인했더니 정말 내 화면이 된다` 수준까지 올라오지 못한다.

### P1. CTA 약속이 상태 전환 후 자연스럽게 이어지지 않음

- 비로그인 CTA는 `오늘의 AI 추천 보기`이고, 로그인 후에는 `전체 분석 보기`, `전략 둘러보기`가 상단 행동으로 제시된다.
- 전환 후 사용자의 기대 행동이 `추천 확인`에서 `분석 탐색/전략 탐색`으로 바뀌어 목적어가 끊긴다.

### P2. 문서 집합 안에 최종 기준 문서가 없음

- 벤치마킹, UX 제안, 구현 갭 분석, 테스트 결과는 있으나 `이 화면이 최종 정답`이라는 canonical 명세는 없다.
- 팀이 문서마다 다른 화면을 상상할 수 있고, 그 결과 구현 우선순위가 흔들릴 위험이 있다.

### P2. 테스트 문서 관리 품질 이슈

- `alphafoundry-homepage-ui-review-test-plan.md`는 동일 내용이 반복 기록되어 있다.
- 검증 문서의 신뢰도를 떨어뜨리고, 후속 리뷰에서 최신 판단 기준을 읽기 어렵게 만든다.

## 문서 검토 관점 핵심 해석

### 1. 홈의 정체성이 문서마다 완전히 닫혀 있지 않다

- `벤치마킹 문서`는 비로그인 홈 개선 중심이다.
- `UX 분석 문서`는 로그인/비로그인 홈의 역할을 거의 분리 수준으로 정의한다.
- `구현 분석 문서`는 Dashboard API 미활용 문제를 정확히 짚지만, 최종 라우트 전략까지 닫지는 못한다.

즉, 문제의식은 맞지만 `그래서 / 는 정확히 누구를 위한 화면인가`가 최종적으로 확정되지 않았다.

### 2. 문서의 방향은 맞지만 구현 반영 수준이 아직 낮다

- `첫 화면 = 데이터`
- `로그인/비로그인은 다른 목적`
- `PC는 multi-column, 모바일은 single-column`
- `Dashboard API 적극 활용`

이 네 가지는 문서들 사이에서 거의 공통 합의다. 실제 화면은 이 합의가 부분적으로만 반영되어 있다.

### 3. 기존 테스트 결과 문서와 현재 실측 사이에 일부 차이가 있다

- 기존 결과 문서에는 `로그인 직후 온보딩 1/3 진입`이 기록돼 있다.
- 이번 실측에서는 `lian_test` 계정 기준으로 온보딩 없이 바로 홈으로 진입했다.

가능한 해석:

- 배포 버전이 바뀌었을 수 있다.
- 계정 상태에 따라 온보딩 노출 조건이 다를 수 있다.
- 이전 테스트는 다른 세션 조건에서 수행됐을 수 있다.

즉, `온보딩 문제`는 현재 시점의 상시 재현 이슈로 단정하기보다, `조건부 재현 가능성`으로 다시 검증하는 편이 안전하다.

## 실제 화면 관찰 요약

### 비로그인

- 첫 화면 카피는 명확하다.
- CTA도 명확하지만, 넓은 화면에서도 모바일형 셸이 유지되어 전문성과 신뢰감이 약해진다.
- 금융/분석 서비스의 첫 화면이라기보다, `모바일 앱 랜딩을 웹에 그대로 올린 느낌`이 강하다.

### 로그인

- 로그인 자체는 정상 동작했다.
- 홈 상단은 `오늘의 AI 분석` 요약형으로 바뀌지만, 이후 흐름은 비로그인과 구조 유사성이 높다.
- 개인화, 긴급성, 시장 맥락, 사용량 같은 `재방문 동기`가 전면에 나오지 않는다.

### PC 관점

- 넓은 폭에서도 정보 밀도 확장이 거의 없었다.
- 문서에서 제안한 `좌측 개인 요약 / 우측 시장+알림 / 하단 병렬 구조`와 차이가 크다.

### 모바일 관점

- 모바일 자체로 보면 현재 셸은 자연스럽지만, `핵심 정보 1st viewport 완결` 원칙은 아직 약하다.
- 로그인 후 첫 화면도 `체크인`보다 `카드 나열형 홈`에 가깝다.

## 왜 메인 화면이 이상하게 느껴지는가

1. 큰 화면에서도 큰 화면답게 보이지 않는다.
2. 로그인 후에도 `내 화면`이 되었다는 느낌이 약하다.
3. 데이터 서비스인데, 핵심 데이터 위계보다 앱 셸 인상이 먼저 들어온다.
4. 문서가 말하는 이상적인 대시보드와 현재 프로덕션의 구조 차이가 크다.
5. 홈의 목적은 바뀌는데, IA와 CTA는 그만큼 선명하게 갈라지지 않는다.

## 우선순위 제안

1. `반응형 구조`부터 바로잡기
   - 데스크톱에서 햄버거 + 하단 탭바 중심 구조를 유지할 이유가 없다면 즉시 수정 검토
   - 큰 화면에서는 상단 전개형 네비게이션과 멀티 컬럼 콘텐츠 구조로 분기

2. `로그인 홈`을 진짜 대시보드로 만들기
   - `signals`, `subscriptions`, `market`, `aiUsage`를 첫 화면 상단 위계에 배치
   - 로그인 직후 사용자가 `오늘 내가 확인할 것`을 즉시 파악하게 만들기

3. `비로그인 홈`은 설득용 랜딩으로 더 날카롭게 정리하기
   - 가치 제안 1개
   - 신뢰 근거 1개
   - 대표 데이터 1개
   - 핵심 CTA 1개

4. `문서 기준점` 하나로 통합하기
   - 최종 화면 책임 문서 1개를 두고
   - 나머지 문서는 근거, 검증, 구현 분석 문서로 역할을 분리

5. `온보딩 재검증` 별도 수행
   - 계정 상태별 노출 조건 정리
   - 신규 가입자/기존 사용자별 진입 흐름 분리 확인

## 권장 후속 액션

- `src/app/page.tsx` 기준으로 로그인/비로그인 홈 책임을 재정의한다.
- `Header`, 하단 탭바, 메인 컨테이너 width/overflow 정책을 먼저 점검한다.
- 문서 기준으로는 `대시보드-UX-전문가-분석.md`를 출발점으로 삼되, 실제 구현 가능한 canonical spec을 별도 확정한다.
